9. Bailout の分類

CalculateFlattenedLengthFast と TryFastFlat の中で goto Bailout が発火する箇所を分類すると以下のとおりです。

分類該当行説明
Map 変異検出行 86、行 217、行 291fastOW.Recheck() otherwise goto Bailout。FastJSArrayForReadWitness.Recheck の中で stable.map != map または IsNoElementsProtectorCellInvalid() を検知します。これは getter コールバック内で Object.setPrototypeOf などにより形が変わるケースを補足するためです。
Length 切り詰め検出行 87、行 218、行 292if (index >= fastOW.Get().length) goto Bailout。getter 内で arr.length = 3 のように長さを縮めた場合に補足します。regress-crbug-1507416.js の TestShrink ケースが該当します。
Smi オーバーフロー行 102、113、122、128、151、行 232、234、306、307math::TrySmiAdd / math::TrySmiSub の otherwise goto Bailout。length 計算と target / source index 進行で使われます。Smi は 32 ビット環境で 31 ビット幅なので非常に大きい結果配列で発火し得ますが、深さ 1024 の制限が事前に効くので実際の発火頻度は低めです。
スタック溢れ行 129、行 308if (stack.length >= kMaxFlatFastStackEntries) goto Bailout。深さ 1024 を超える入れ子で発火します。サイクル a.push(a) がこれを引きます。
Cast 失敗系行 77、99、111、120、137、167-168、187、189、208、230、242、269-270、282、304、316、341-343多岐にわたります。Cast<FastJSArrayForRead>(source)、子要素 Cast<FastJSArrayForRead>(element)、Cast<Smi>(elementArray.length)、currentLength、スタックポップ時の再キャスト、Cast<Smi>(sourceLength)、Cast<FastJSArrayForCopy>(receiver) など。走査中に ElementsKind が遷移して FastJSArray* 型ガードを満たさなくなった、または length が Smi 範囲外になった、もしくは入力 receiver が proxy だった、などのケースを捕捉します。
Proxy 専用の早期 bailout行 143、248、322if (currentDepth > 0 && Is<JSProxy>(element)) goto Bailout。depth > 0 でかつ要素がプロキシだと、その要素は IsArray で配列に化ける可能性があり (runtime::ArrayIsArray は proxy 内部のターゲットを判定する)、fast path では proxy のトラップを発火させたくないため slow path に投げます。depth == 0 のときは要素はそのままコピーされるだけで配列扱いされないので、proxy でも bailout しません。
配列長一致の最終整合性確認行 274、行 348if (targetIndex != flattenedLength) goto Bailout。第一パスで計算した長さと第二パスで実際に書き込んだ件数が一致しなかった場合の最後の安全弁です。
Double 配列第二パスの境界チェック行 249-250if (Convert<intptr>(targetIndex) >= doubleElements.length_intptr) goto Bailout
通常パスの境界チェック行 323-324if (Convert<intptr>(targetIndex) >= vector.fixedArray.length_intptr) goto Bailout

FlattenIntoArrayFast 側 (行 352-435) の Bailout は (Number, Number) 引数を取って targetIndex と smiSourceIndex を呼び出し元に返し、FlattenIntoArray の label Bailout (行 508-512) で受け取ってその進捗から slow path を再開する設計になっており、走査済みの要素を再走査しないでよくする工夫です。発火点は行 363 (source が FastJSArray でない)、行 372 (Recheck 失敗)、行 376 (length 切り詰め) です。
